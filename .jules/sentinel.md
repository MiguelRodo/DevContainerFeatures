## 2024-05-18 - [CRITICAL] Command Injection in mermaid-mmdc wrapper
**Vulnerability:** The wrapper script `src/mermaid/cmd/mmdc` constructed a command string `CMD="mmdc -p \"$CONFIG_FILE\" $*"` which combined all user-supplied arguments (`$*`) into a single string. This string was then passed to `bash -c "$CMD"`, allowing an attacker to pass crafted arguments (like `"; malicious_command; "`) to achieve arbitrary command execution as another user.
**Learning:** Shell scripts that wrap other commands and drop privileges using `su` or `sudo` must be extremely careful to avoid evaluating user-supplied arguments in a shell context. Expanding arguments using `$*` and passing them to `bash -c` is inherently vulnerable to shell injection.
**Prevention:** When passing arguments through `su` or `sudo`, always preserve them as arrays using `"$@"`. For `sudo`, commands can often be executed directly with `sudo -u "$USER" command "$@"`. For `su`, safely passing an array can be achieved by writing `su -s /bin/bash "$USER" -c 'exec command "$0" "$@"' -- "$arg1" "$@"` where the bash `-c` script executes the command, passing the positional parameters correctly.
## 2024-05-20 - [CRITICAL] Arbitrary File Deletion / Path Traversal via `rm -rf $DIR/*`
**Vulnerability:** A shell script used `rm -rf "$directory"/*` as a fallback mechanism to empty a directory. If the `$directory` variable was empty or unset, this evaluated to `rm -rf /*`, which would destroy the container's entire filesystem. Similarly, if `$directory` was unexpectedly set to `/`, it would have the same effect.
**Learning:** Shell scripts frequently perform file cleanup and directory manipulation tasks. Relying purely on string interpolation `"$DIR"/*` for destructive commands like `rm -rf` is extremely hazardous without strict validation. This vulnerability pattern is common when cleaning up temporary directories and can silently wipe the filesystem if variable initialization fails or arguments are passed incorrectly.
**Prevention:** Always validate directory paths before performing destructive operations. Check that `$DIR` is not empty (`[ -z "$DIR" ]`) and is not the root path (`[ "$DIR" = "/" ]`). Additionally, leverage Bash's variable validation syntax `rm -rf "${DIR:?}"/*` which immediately aborts execution and throws an error if `DIR` is unset or empty, preventing the catastrophic expansion to `/*`.
## 2025-03-10 - [CRITICAL] Arbitrary File Deletion / Path Traversal via unvalidated `rm -rf` commands
**Vulnerability:** Several shell scripts used unvalidated string interpolation in destructive commands, such as `rm -rf "$TEMP_DIR"` and `rm -rf "$dir"`. If the variables were unexpectedly empty or set to root, this evaluated to `rm -rf ""` or `rm -rf "/"`, which can cause devastating data loss or destroy the entire filesystem. The same issue occurred with missing variables in patterns like `rm -rf /tmp/fftw-${FFTW_VERSION}`.
**Learning:** Shell scripts frequently perform file cleanup tasks and must be robust against unset variables. Simple parameter expansions are dangerous if their value isn't guaranteed. Relying on string interpolation without explicit emptiness and path checks enables severe directory traversal vulnerabilities when variables fail to initialize.
**Prevention:** Always validate parameters before destructive operations. First, use bash parameter expansion validation syntax such as `rm -rf "${DIR:?}"` which aborts execution immediately if `DIR` is unset or empty. Second, explicitly check for critical directories (`if [ -z "$dir" ] || [ "$dir" = "/" ]; then ...`) when deleting paths provided by external arguments or complex logic to avoid unintentional file erasure.
## 2024-05-24 - [Command Injection via DevContainer Feature Options]
**Vulnerability:** DevContainer feature options passed as unvalidated strings were written to an `.env` file via `cat << EOF` blocks, which was later `source`d on shell startup (`~/.bashrc.d/`). This allowed for arbitrary command injection if an attacker supplied a malicious string.
**Learning:** Writing arbitrary environment variables directly into a script or `.env` file intended to be evaluated by `source` without proper sanitization/validation results in command injection.
**Prevention:** Strictly validate or sanitize inputs (e.g. restrict to `true` or `false` for booleans) or use safe serialization (like `printf '%q'`) before writing values to any file that will be dynamically executed or sourced.
## 2025-01-01 - [Code Injection via renv.lock]
**Vulnerability:** In `src/renv-cache/cmd/renv-restore`, the version parsed from `renv.lock` was directly interpolated into a single-quoted R string (`target <- '${target_version}'`) within `Rscript -e`. Since `renv.lock` is a user-controlled project file, a malicious attacker could put an R payload in the Version field (e.g., `1.0.0'; system('id'); '`), leading to arbitrary R code execution when the container restores packages.
**Learning:** String interpolation of variables derived from user-controlled files directly into code strings (especially scripts passed to `Rscript -e`, `python -c`, `bash -c`, etc.) is a common vector for code and command injection.
**Prevention:** Always pass external input securely to execution environments, such as through environment variables (`Sys.getenv` in R) or secure argument arrays, instead of manipulating string templates.
## 2024-04-07 - [CRITICAL] Privilege Escalation via Interactive Shell for System Users
**Vulnerability:** In `src/mermaid/install.sh`, a non-root system user was created with an interactive shell (`useradd --system --shell /bin/bash "$USERNAME"` and `adduser -D -s /bin/sh "$USERNAME"`). This violates the principle of least privilege, as these system users do not need interactive shell access. If the user account is compromised, the attacker would gain full interactive access.
**Learning:** Creating system accounts with interactive shells like `/bin/bash` or `/bin/sh` unnecessarily increases the attack surface. In a DevContainer environment, dedicated service or system users should be tightly locked down to prevent lateral movement.
**Prevention:** Always create non-root system users with a restricted shell (e.g., `/sbin/nologin` or `/bin/false`). When execution as these users is required during setup, explicitly invoke a valid shell using `su` (e.g., `su -s /bin/bash - "$USERNAME" -c '...'`).
## 2025-05-24 - [CRITICAL] Command Injection via unvalidated Node.js version input
**Vulnerability:** In `src/mermaid/install.sh`, an unvalidated environment variable `NODE_VERSION` was used directly in a `curl` command piped to bash (`curl -fsSL "https://deb.nodesource.com/setup_${NODE_VERSION}.x" | bash -`) and in a `dnf` command (`dnf module enable -y "nodejs:${NODE_VERSION}"`). A malicious user could provide a crafted `NODE_VERSION` (e.g., `20"; echo "PWNED"`) to achieve arbitrary command execution as root during container build.
**Learning:** DevContainer feature options passed as environment variables must be rigorously validated before being used in shell commands, especially when interpolated into strings that are evaluated or executed.
**Prevention:** Always use strict validation (e.g., regex `^[a-zA-Z0-9._-]+$`) for variables derived from user input or feature options before using them in commands.
## 2025-10-18 - [CRITICAL] Command Injection Bypass due to Bashisms in Strict POSIX Environments
**Vulnerability:** Security validation checks across multiple shell scripts relied on bash-specific syntax (`[[ "$VAR" == -* ]]` or `=~`) despite being executed in potentially strict POSIX shell environments (like `/bin/sh` symlinked to `dash` in Debian/Ubuntu DevContainers). Additionally, the regex for `USERNAME` permitted arbitrary characters as long as it did not start with a dash.
**Learning:** In strict POSIX shells, bashisms like `[[` result in a 'not found' command error. If placed in an `if` statement meant to block malicious input, the error causes the condition to evaluate to false (failing open), entirely bypassing the security control and enabling command injection. Additionally, checking for a leading dash only prevents option injection, not full command injection.
**Prevention:** Always use POSIX-compliant constructs such as `echo "$VAR" | grep -Eq '^[pattern]$'` for input validation in DevContainer setup scripts. Furthermore, when validating user input that will interact with system commands, use strict allowlist regular expressions (e.g., `^[a-zA-Z0-9_][a-zA-Z0-9_-]*$`) rather than weak deny-list checks.
## 2025-05-25 - [CRITICAL] Command and Option Injection Bypass via Weak Deny-list in git checkout
**Vulnerability:** In `src/fit-sne/install.sh`, the version string from the `FITSNE_VERSION` variable was validated using a weak deny-list regex: `echo "${FITSNE_VERSION}" | grep -Eq '^-'`. This only prevented arguments starting with a dash, but allowed arbitrary characters like spaces, semicolons, and quotes. This could allow an attacker to pass arbitrary commands or bypass the option validation if passed into the subsequent `git checkout` command.
**Learning:** Deny-lists (like checking if an argument starts with `-`) are inherently weak and fail to account for the full spectrum of shell injection vectors. They do not ensure the input strictly conforms to the expected structure.
**Prevention:** Always use strict, POSIX-compliant allow-list regular expressions to validate user-supplied options and variables before using them in commands. For versions, this might look like `echo "${FITSNE_VERSION}" | grep -Eq '^[a-zA-Z0-9_][a-zA-Z0-9_.-]*$'`.
## 2024-05-01 - CmdStan Command Injection
**Vulnerability:** A command injection vulnerability was found in the CmdStan feature installation script (`src/cmdstan/install.sh`). The unvalidated option `INSTALL_DIR` was interpolated directly into `/etc/profile.d/cmdstan.sh` and `/etc/environment` using string substitution in bash. An attacker could supply an unsafe value (e.g., `/opt/cmdstan"; touch /tmp/pwned; #`) allowing them to execute arbitrary code with root privileges during the build process, or corrupt system paths.
**Learning:** DevContainer feature scripts are susceptible to command injection and path corruption when user-provided configuration options (which may be loaded dynamically from `devcontainer-feature.json`) are insecurely written directly into bash profiles or configuration files.
**Prevention:** Always validate all external options (paths, usernames, booleans, versions) against strict POSIX-compliant regex patterns (e.g., `echo "$VAR" | grep -Eq '^[pattern]$'`) before using them in scripts or writing them to files. Specifically, directory paths must be validated to ensure they are absolute and contain only safe characters.
