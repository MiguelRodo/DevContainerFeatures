## 2024-05-18 - [CRITICAL] Command Injection in mermaid-mmdc wrapper
**Vulnerability:** The wrapper script `src/mermaid/cmd/mmdc` constructed a command string `CMD="mmdc -p \"$CONFIG_FILE\" $*"` which combined all user-supplied arguments (`$*`) into a single string. This string was then passed to `bash -c "$CMD"`, allowing an attacker to pass crafted arguments (like `"; malicious_command; "`) to achieve arbitrary command execution as another user.
**Learning:** Shell scripts that wrap other commands and drop privileges using `su` or `sudo` must be extremely careful to avoid evaluating user-supplied arguments in a shell context. Expanding arguments using `$*` and passing them to `bash -c` is inherently vulnerable to shell injection.
**Prevention:** When passing arguments through `su` or `sudo`, always preserve them as arrays using `"$@"`. For `sudo`, commands can often be executed directly with `sudo -u "$USER" command "$@"`. For `su`, safely passing an array can be achieved by writing `su -s /bin/bash "$USER" -c 'exec command "$0" "$@"' -- "$arg1" "$@"` where the bash `-c` script executes the command, passing the positional parameters correctly.
## 2024-05-20 - [CRITICAL] Arbitrary File Deletion / Path Traversal via `rm -rf $DIR/*`
**Vulnerability:** A shell script used `rm -rf "$directory"/*` as a fallback mechanism to empty a directory. If the `$directory` variable was empty or unset, this evaluated to `rm -rf /*`, which would destroy the container's entire filesystem. Similarly, if `$directory` was unexpectedly set to `/`, it would have the same effect.
**Learning:** Shell scripts frequently perform file cleanup and directory manipulation tasks. Relying purely on string interpolation `"$DIR"/*` for destructive commands like `rm -rf` is extremely hazardous without strict validation. This vulnerability pattern is common when cleaning up temporary directories and can silently wipe the filesystem if variable initialization fails or arguments are passed incorrectly.
**Prevention:** Always validate directory paths before performing destructive operations. Check that `$DIR` is not empty (`[ -z "$DIR" ]`) and is not the root path (`[ "$DIR" = "/" ]`). Additionally, leverage Bash's variable validation syntax `rm -rf "${DIR:?}"/*` which immediately aborts execution and throws an error if `DIR` is unset or empty, preventing the catastrophic expansion to `/*`.
## 2025-03-10 - [CRITICAL] Arbitrary File Deletion / Path Traversal via unvalidated `rm -rf` commands
**Vulnerability:** Several shell scripts used unvalidated string interpolation in destructive commands, such as `rm -rf "$TEMP_DIR"` and `rm -rf "$dir"`. If the variables were unexpectedly empty or set to root, this evaluated to `rm -rf ""` or `rm -rf "/"`, which can cause devastating data loss or destroy the entire filesystem. The same issue occurred with missing variables in patterns like `rm -rf /tmp/fftw-${FFTW_VERSION}`.
**Learning:** Shell scripts frequently perform file cleanup tasks and must be robust against unset variables. Simple parameter expansions are dangerous if their value isn't guaranteed. Relying on string interpolation without explicit emptiness and path checks enables severe directory traversal vulnerabilities when variables fail to initialize.
**Prevention:** Always validate parameters before destructive operations. First, use bash parameter expansion validation syntax such as `rm -rf "${DIR:?}"` which aborts execution immediately if `DIR` is unset or empty. Second, explicitly check for critical directories (`if [ -z "$dir" ] || [ "$dir" = "/" ]; then ...`) when deleting paths provided by external arguments or complex logic to avoid unintentional file erasure.
